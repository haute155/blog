SSG, SSR, CSR, ISR, 하이드레이션, RSC, 엣지 렌더링. 프론트엔드를 하다 보면 이 용어들을 각각은 설명할 수 있게 됩니다. 그런데 이것들이 서로 어떤 관계인지, 왜 이 순서로 등장했는지를 물으면 대답이 흐려지는 경우가 많습니다. 각 개념을 따로 배웠기 때문입니다.

이 글은 그 파편들을 하나의 질문 위에 재배치하려는 시도입니다.

> **HTML은 누가, 언제, 어디서 만드는가?**

모든 렌더링 전략은 이 질문에 대한 답이고, 모든 "최신 기법"은 이전 답의 한계에 대한 반작용입니다. 계보를 따라가 봅시다.

## 출발점: 서버의 일은 두 종류다

브라우저가 `GET /post/1`을 보내면 서버는 HTML을 응답해야 합니다. 이때 서버가 하는 일은 본질적으로 둘 중 하나입니다.

**파일을 읽어서 보내거나** (정적) — 디스크에 이미 존재하는 바이트를 소켓으로 밀어 넣는 I/O 작업입니다. 커널의 `sendfile` 같은 시스템 콜은 유저 공간을 거치지도 않고 파일을 네트워크로 복사합니다. CPU가 하는 일이 거의 없습니다.

**프로그램을 실행해서 만들거나** (동적) — 런타임을 깨우고, DB에 쿼리하고, 템플릿을 조립하는 CPU + I/O 복합 작업입니다. 요청마다 이 과정이 반복됩니다.

이 구분이 중요한 이유는 **확장 비용의 곡선이 다르기 때문**입니다.

|                    | 정적 파일 서빙           | 동적 렌더링                 |
| ------------------ | ------------------------ | --------------------------- |
| 요청당 비용        | I/O (거의 0에 수렴)      | CPU + 메모리 + DB 커넥션    |
| 트래픽 10배가 되면 | 회선만 늘리면 됨         | 컴퓨팅을 10배로 늘려야 함   |
| 복제 가능성        | 전 세계 어디든 복사 가능 | 상태(DB)와 함께 움직여야 함 |

정적 파일의 결정적 성질은 **내용이 요청과 무관하게 동일하다**는 것입니다. 동일하므로 복사할 수 있고, 복사할 수 있으므로 CDN의 수백 개 엣지 노드에 뿌려둘 수 있습니다. 서울의 방문자는 서울 PoP에서, 프랑크푸르트의 방문자는 프랑크푸르트 PoP에서 응답을 받습니다. 원본 서버(origin)는 캐시가 만료될 때만 일합니다.

호스팅 업체들이 정적 사이트를 무료로 무제한에 가깝게 제공하는 이유가 여기 있습니다. 정적 서빙의 한계비용은 0에 수렴하고, 남의 코드를 실행하지 않으므로 무한루프나 채굴 같은 리스크도 없습니다. 반면 동적 렌더링은 요청 하나하나가 업체의 CPU를 태우는 일이라, 무료로 풀 때는 반드시 실행 횟수·시간 제한이 붙습니다.

<Callout type="info" title="캐시는 공짜가 아니다">

  입니다. 글을 수정했는데 어느 엣지 노드에는 옛 버전이 남아 있다면? TTL,{" "}
  <code>stale-while-revalidate</code>, 퍼지 API, 캐시 키 설계 같은 도구가 전부
  이 문제를 다루기 위해 존재합니다. "컴퓨터 과학의 두 가지 난제는 캐시 무효화와
  이름 짓기"라는 농담은 여기서도 유효합니다.
</Callout>

## 1세대 답안 셋: SSG, SSR, CSR

"HTML을 누가, 언제 만드는가"에 대한 고전적인 세 가지 답과, 각각이 부딪힌 한계입니다.

### SSG — 빌드 타임에, 빌드 머신이

배포 전에 모든 페이지를 HTML로 구워두는 방식입니다. 요청 시점에는 위에서 말한 "파일 읽어서 보내기"만 남으므로 TTFB가 가장 짧고, 서버 장애라는 개념 자체가 희미해집니다. 실행될 코드가 없으니 공격 표면도 최소입니다.

한계는 두 가지입니다. **콘텐츠 갱신에 재빌드가 필요하다**는 것, 그리고 **빌드 시간이 페이지 수에 비례**한다는 것. 페이지가 100개일 때는 아무 문제가 없지만, 상품 10만 개짜리 커머스라면 빌드 한 번에 수십 분이 걸립니다. 오타 하나 고치려고 10만 페이지를 다시 굽는 것은 명백히 잘못된 구조입니다.

### SSR — 요청 타임에, 서버가

매 요청마다 서버가 데이터를 조회해 HTML을 조립합니다. 항상 최신이고, 요청자(로그인 사용자, 지역, 디바이스)에 따라 다른 내용을 만들 수 있습니다.

비용은 명확합니다. 렌더링이 응답 경로(critical path) 위에 놓이므로 **DB가 느리면 TTFB가 함께 느려지고**, 트래픽 스파이크는 곧 컴퓨팅 스파이크입니다. 그리고 뒤에서 다룰 **하이드레이션**이라는 숨은 청구서가 따라옵니다.

### CSR — 접속 후에, 방문자의 브라우저가

서버는 거의 빈 HTML과 자바스크립트 번들만 주고, 브라우저가 API를 호출해 화면을 조립합니다. 2010년대 SPA 전성기의 방식이고, 한 번 로드된 뒤의 화면 전환은 앱처럼 매끄럽습니다.

문제는 **첫 화면까지의 거리**입니다. HTML 파싱 → JS 다운로드 → 파싱/실행 → API 호출 → 렌더링이라는 직렬 체인을 다 통과해야 콘텐츠가 보입니다. 번들은 기능이 늘수록 커지는데, 이 비용은 서버가 아니라 **방문자의 기기**가 냅니다. 저사양 폰에서 체감이 가장 나쁜 이유입니다. 검색엔진 크롤러의 JS 렌더링 예산 문제까지 겹치면, 콘텐츠 사이트에서 CSR은 선택지에서 빠집니다.

|     | 만드는 곳       | 만드는 시점 | 강점               | 아킬레스건                    |
| --- | --------------- | ----------- | ------------------ | ----------------------------- |
| SSG | 빌드 머신       | 배포 전     | 속도, 비용, 안정성 | 재빌드, 빌드 시간 폭발        |
| SSR | 서버            | 매 요청     | 최신성, 개인화     | 서버 비용, TTFB, 하이드레이션 |
| CSR | 방문자 브라우저 | 접속 후     | 앱 같은 전환       | 초기 로드, SEO, 기기 부담     |

## 숨은 비용: 하이드레이션

SSR을 이해했다고 말하려면 이 질문에 답할 수 있어야 합니다. **"서버가 HTML을 다 만들어 보냈는데, 왜 클라이언트에 또 React가 필요한가?"**

서버가 보낸 것은 완성된 그림입니다. 버튼이 _보이지만_ 클릭 핸들러는 없습니다. 이벤트 리스너를 붙이려면 클라이언트에서 컴포넌트 트리를 다시 실행해 가상 DOM을 만들고, 서버가 보낸 실제 DOM과 짝을 맞춰야 합니다. 이 과정이 **하이드레이션**입니다.

여기서 두 가지 비효율이 발생합니다.

1. **이중 실행** — 같은 컴포넌트 코드가 서버에서 한 번(HTML 생성), 클라이언트에서 또 한 번(이벤트 연결) 실행됩니다.
2. **이중 전송** — HTML 안에 렌더된 데이터가, 하이드레이션을 위해 `<script>` 안의 직렬화된 JSON으로 **한 번 더** 실려 옵니다. SSR 페이지의 소스를 열어 거대한 JSON 덩어리를 본 적이 있다면 그것입니다.

그 결과가 **uncanny valley**입니다: 화면은 보이는데(FCP는 빠른데) 몇 초간 아무것도 눌리지 않는 구간(TTI 지연). SSR은 "보이는 속도"를 해결했지만 "동작하는 속도"는 CSR의 문제를 그대로 물려받은 셈입니다.

**이후 등장한 거의 모든 기법은 이 하이드레이션 비용을 줄이려는 시도로 읽을 수 있습니다.**

## 2세대: 한계에 대한 반작용들

### ISR — SSG의 재빌드 문제를 공략

Incremental Static Regeneration. "정적으로 굽되, N초가 지난 뒤 요청이 오면 일단 옛 버전을 주고 백그라운드에서 다시 굽는다"는 방식입니다. HTTP 캐시의 `stale-while-revalidate` 의미론을 페이지 생성에 적용한 것으로, 10만 페이지 커머스도 "자주 팔리는 상품만 갱신되는 정적 사이트"가 될 수 있습니다. 정적과 동적 사이의 첫 번째 다리입니다.

### 스트리밍 SSR — TTFB 문제를 공략

렌더링이 다 끝나야 응답을 시작하는 대신, **완성되는 부분부터 청크로 흘려보냅니다.** `<head>`와 레이아웃을 먼저 보내 브라우저가 CSS·폰트를 미리 받게 하고, 느린 DB 쿼리에 묶인 영역은 placeholder를 먼저 보낸 뒤 나중에 채웁니다. React 18의 `<Suspense>` + `renderToPipeableStream`이 이 모델입니다. "서버가 느리면 전부 느리다"는 SSR의 약점을 "느린 부분만 늦게"로 완화합니다.

### 아일랜드 아키텍처 — 하이드레이션 범위를 공략

"페이지 전체를 하이드레이션할 필요가 있는가?"라는 질문에서 출발합니다. 콘텐츠 페이지의 대부분은 정적 HTML로 충분하고, 인터랙션이 필요한 곳은 검색창·장바구니 버튼 정도입니다. 그렇다면 **그 섬(island)들만 JS를 보내고 하이드레이션하면 됩니다.** Astro가 이 모델의 대표 구현이고, 기본값이 "JS 0"이라는 점이 핵심입니다.

### Resumability — 하이드레이션 자체를 제거

Qwik이 제시한 급진적인 답입니다. 서버가 렌더링하면서 **이벤트 핸들러의 위치와 상태를 HTML 속성에 직렬화**해두면, 클라이언트는 트리를 재실행할 필요 없이 그 지점에서 "이어서 실행(resume)"할 수 있습니다. 클릭이 일어나는 순간 해당 핸들러 코드만 lazy 로드합니다. 이중 실행과 이중 전송을 구조적으로 제거한 설계로, 채택과 별개로 하이드레이션이 필연이 아님을 증명했다는 의미가 있습니다.

## 3세대: 단위가 페이지에서 컴포넌트로

여기까지의 기법들은 여전히 "이 **페이지**는 SSG인가 SSR인가"를 묻습니다. 최근의 전환은 그 단위 자체를 바꿉니다.

### React Server Components

RSC의 핵심 계약은 이것입니다: **서버 컴포넌트의 코드는 클라이언트 번들에 포함되지 않는다.**

```tsx
// 서버 컴포넌트 — 이 코드도, marked 라이브러리도 브라우저에 안 간다
async function PostBody({ id }: { id: string }) {
  const post = await db.post.findUnique({ where: { id } }); // 직접 DB 접근
  return <article>{renderMarkdown(post.body)}</article>;
}

// 클라이언트 경계 — 여기서부터만 번들에 포함
("use client");
function LikeButton() { ... }
```

서버 컴포넌트는 렌더링 결과만 **RSC 페이로드**(직렬화된 트리)로 내려보내고, 클라이언트 컴포넌트만 하이드레이션됩니다. 마크다운 파서, DB 클라이언트, 무거운 날짜 라이브러리가 번들에서 사라집니다. "이 페이지는 SSR인가"라는 질문이 "이 **컴포넌트**는 어느 쪽 경계에 있는가"로 바뀌는 것입니다.

아일랜드와 비교하면 방향이 반대인 점이 흥미롭습니다. 아일랜드는 "정적 바다 위에 인터랙티브 섬"이고, RSC는 "서버 트리 안에 클라이언트 가지"입니다. 출발점은 다르지만 두 모델의 결론은 같습니다 — **기본값은 서버, 클라이언트 JS는 명시적으로 선택한 곳에만.**

### Partial Prerendering — 정적과 동적의 동거

한 페이지를 빌드 타임에 분석해서, 정적인 껍데기는 CDN에서 즉시 서빙하고 동적인 구멍(`<Suspense>` 경계)만 서버가 스트리밍으로 채우는 모델입니다. 커머스 상품 페이지로 예를 들면:

```text
상품명, 이미지, 설명   → 정적 (빌드 타임, CDN 캐시)
가격, 재고            → 반동적 (ISR 또는 짧은 캐시)
"OO님을 위한 추천"     → 동적 (요청 타임, 스트리밍)
장바구니 배지          → 클라이언트 (브라우저에서 조회)
```

하나의 URL 안에 네 가지 렌더링 전략이 공존합니다. "정적 vs 동적"이 사이트의 속성에서 페이지의 속성으로, 다시 **컴포넌트의 속성**으로 내려온 과정의 종착점입니다.

### 엣지 런타임 — "어디서"의 재정의

전통적 서버리스(AWS Lambda)는 요청이 오면 컨테이너를 깨웁니다. 콜드 스타트가 수백 ms에 달할 수 있고, 리전은 몇 곳뿐입니다. Cloudflare Workers류의 엣지 런타임은 다른 구조를 씁니다 — 컨테이너 대신 **V8 isolate**. 하나의 프로세스 안에서 격리된 컨텍스트를 띄우는 방식이라 콜드 스타트가 밀리초 단위이고, 수백 개 PoP 전부에서 실행됩니다.

이것이 의미하는 바: **동적 렌더링을 CDN 캐시가 있는 바로 그 자리에서 돌릴 수 있게 됐습니다.** "정적=엣지=빠름 / 동적=원본=느림"이라는 지리적 등식이 깨진 것입니다. 대가는 있습니다. isolate는 Node.js가 아니므로 파일시스템, 네이티브 모듈, 일부 Node API를 쓸 수 없고, 데이터가 여전히 한 리전의 DB에 있다면 엣지에서 DB까지의 왕복이 새로운 병목이 됩니다. 엣지 렌더링의 진짜 과제는 컴퓨팅이 아니라 **데이터의 지리적 분산**입니다.

<Callout type="warning" title="엣지가 항상 답은 아니다">
  렌더링을 사용자 근처로 옮겨도 DB가 버지니아에 하나뿐이라면, 서울 엣지에서
  실행되는 SSR은 요청마다 태평양을 왕복합니다. 원본 근처에서 렌더링하고 CDN으로
  캐싱하는 편이 빠른 경우도 많습니다. 엣지 컴퓨팅과 데이터 복제(edge DB, 리드
  레플리카)는 세트로 고민해야 하는 주제입니다.
</Callout>

### 반대 방향: 단순화 진영

복잡도를 프레임워크가 흡수하는 RSC 노선의 반대편에는, 복잡도 자체를 거부하는 흐름이 있습니다. HTMX는 "서버가 HTML 조각을 반환하고 클라이언트는 그것을 갈아끼우기만 한다"는 하이퍼미디어 모델로 회귀합니다. 90년대의 서버 렌더링과 다른 점은 부분 갱신이 가능하다는 것뿐인데, 그것만으로 상당수의 SPA를 대체할 수 있다는 것이 이 진영의 주장입니다.

두 진영은 방법론에서 갈리지만 전제를 공유합니다. **브라우저로 보내는 자바스크립트는 부채이며, 갚을 이유가 있을 때만 빌려야 한다.** 2010년대 CSR 전성기에 대한 업계 전체의 반성문인 셈입니다.

## 정리: 판별 프레임워크

계보를 요약하면 이렇게 됩니다.

```text
정적 vs 동적 (사이트 단위)
  → SSG / SSR / CSR (페이지 단위)
    → ISR, 스트리밍, 아일랜드 (페이지 내 시점·범위 분할)
      → RSC, PPR (컴포넌트 단위)
         + 엣지 (실행 위치의 재배치)
```

실무에서의 판별은 화면의 각 영역에 두 가지 질문을 던지는 것으로 충분합니다.

1. **이 내용은 보는 사람마다 다른가?** — 아니라면 캐시 가능한 영역입니다. 정적이나 ISR로 밀어내세요.
2. **이 내용은 언제 바뀌는가?** — 배포할 때만(SSG), 주기적으로(ISR), 매 요청마다(SSR/스트리밍), 사용자 행동으로(클라이언트).

그리고 콘텐츠 중심 사이트 — 블로그, 문서, 마케팅 페이지 — 에 한정하면 답은 여전히 명확합니다. 모두에게 같고, 배포할 때만 바뀌고, 크롤러에 잘 읽혀야 하는 콘텐츠에는 SSG + 최소한의 클라이언트 JS가 정답이고, 이 답은 RSC 시대에도 낡지 않았습니다. 최신 기법들이 하려는 일이 결국 "동적인 서비스를 최대한 정적인 것처럼 보이게 만들기"라는 점을 생각하면, 처음부터 정적인 콘텐츠는 이미 종착점에 서 있는 셈입니다.
