개발을 하다 보면 터미널 명령어를 하나씩 주워 담게 됩니다. `lsof -ti :3000 | xargs kill`을 복사해 붙여넣고, 동작하니 넘어갑니다. 그런데 이 명령이 **왜** 이렇게 생겼는지 설명할 수 있는 사람은 의외로 적습니다. "파일 목록을 보여준다"는 `lsof`가 왜 네트워크 포트를 보여줄까요? `kill` 앞에 붙는 `xargs`는 대체 뭘 하는 걸까요?

이 글은 명령어 모음이 아닙니다. 그 밑에 깔린 **모델**을 세우는 글입니다. 모델이 서면 명령어는 외우는 것이 아니라 유도되는 것이 됩니다.

## 터미널, 셸, OS — 층부터 구분하자

"터미널을 배운다"는 말부터 해부해 봅시다. 검은 창에 명령을 치는 행위에는 사실 네 개의 층이 겹쳐 있습니다.

```text
① 터미널       — 입출력 창. 키 입력을 전달하고 글자를 그린다
② 셸 (zsh, bash) — 명령어 해석기. 텍스트를 해석해 OS에 요청한다
③ OS (유닉스/리눅스) — 실제로 일하는 커널
④ 하드웨어
```

**터미널**은 원래 물리 장치였습니다. 중앙 컴퓨터에 연결된 말단(terminal)의 키보드와 화면이 그 어원이고, 지금 쓰는 앱들은 그 장치를 소프트웨어로 흉내 낸 "터미널 에뮬레이터"입니다. **셸**은 커널(kernel, 알맹이)을 감싸는 껍데기(shell)라서 셸입니다. 사용자의 텍스트를 해석해서 커널에 시스템 콜로 전달하는 통역자입니다.

식당에 비유하면 터미널은 테이블과 메뉴판, 셸은 주문을 받는 웨이터, OS는 주방입니다. 이 층 구분이 서면 흔한 의문들이 저절로 풀립니다.

- 터미널 앱을 바꿔도(iTerm2, Alacritty, 기본 Terminal) 명령어가 똑같이 동작하는 이유 — ①층만 바뀌었기 때문
- macOS에서 배운 명령이 리눅스 서버에서 통하는 이유 — ③층이 같은 유닉스 계열이기 때문
- zsh과 bash에서 스크립트 동작이 미묘하게 다른 이유 — ②층이 다르기 때문

즉 "터미널을 배운다"는 것의 실체는 **셸의 문법과 유닉스의 도구·개념을 배우는 것**입니다. 검은 창 자체는 배울 것이 거의 없습니다.

## 모든 것은 파일이다

유닉스 설계의 대원칙이자, 수많은 명령어의 생김새를 설명해 주는 열쇠입니다.

유닉스는 디스크의 문서뿐 아니라 **거의 모든 자원을 파일이라는 단일 인터페이스로 추상화**합니다. 키보드 입력도 파일(stdin), 화면 출력도 파일(stdout), 하드웨어 장치도 파일(`/dev/*`), 프로세스 정보도 파일(`/proc/*`), 그리고 **네트워크 소켓도 파일**입니다. 어떤 자원이든 열고(open), 읽고(read), 쓰고(write), 닫는(close) 동일한 동사로 다룰 수 있습니다.

이 원칙을 알고 나면 대표적인 수수께끼가 풀립니다. 개발 서버를 띄우려는데 포트가 이미 점유되어 있을 때 쓰는 명령:

```bash
lsof -i :3000
```

`lsof`는 **LiSt Open Files**, 말 그대로 "열린 파일 목록"입니다. 파일 목록 명령이 포트 점유자를 보여주는 이유는 간단합니다 — 소켓도 파일이므로, "3000번 포트를 누가 쓰고 있나?"는 곧 "그 소켓 파일을 어떤 프로세스가 열고 있나?"라는 질문이기 때문입니다. `-i`는 **i**nternet 소켓으로 범위를 좁히는 옵션입니다.

출력에서 확인할 것은 PID(프로세스 식별 번호)와 COMMAND(정체)입니다. 유닉스에서 모든 프로세스는 PID로 식별되고, PID를 향해 시그널을 보낼 수 있습니다.

```bash
kill 11524      # SIGTERM — "정리하고 종료해라" (기본값)
kill -9 11524   # SIGKILL — 커널이 즉시 제거. 프로세스에게 기회를 주지 않는다
```

`kill`이라는 이름과 달리 실체는 "시그널 전송"입니다. 기본 시그널인 SIGTERM은 프로세스에게 뒷정리(파일 저장, 커넥션 종료)할 기회를 주고, `-9`(SIGKILL)는 그 기회 없이 즉사시킵니다. 항상 SIGTERM을 먼저 시도하고, 반응이 없을 때만 `-9`로 올라가는 것이 순서입니다.

<Callout type="info" title="같은 개념, 다른 도구">
  리눅스 서버에서는 <code>lsof</code> 대신 <code>ss -tlnp</code>를 쓰기도
  합니다. 도구는 달라도 "소켓을 점유한 프로세스를 찾아 PID로 제어한다"는 모델은
  동일합니다. 모델을 알면 도구가 바뀌어도 당황할 일이 없습니다.
</Callout>

## 파이프 — 프로그램을 문장처럼 조립하기

유닉스 철학의 두 번째 기둥은 **"하나의 도구는 하나의 일만 하고, 텍스트로 소통한다"**입니다. 각 도구를 작고 단순하게 유지하는 대신, 파이프(`|`)로 이어 붙여 복잡한 작업을 조립합니다. 앞 명령의 출력이 뒤 명령의 입력이 되는 구조입니다.

포트 정리를 한 줄로 끝내는 관용구가 좋은 예입니다.

```bash
lsof -ti :3000 | xargs kill
```

- `-t`(**t**erse)는 출력을 PID만으로 줄입니다 — 사람용 표 대신 기계용 텍스트
- 그 PID가 파이프를 타고 다음 명령으로 흐릅니다

그런데 여기서 `xargs` 없이 `| kill`로 바로 이으면 실패합니다. 이 지점이 파이프를 이해했는지 가르는 시험대입니다.

```bash
lsof -ti :3000 | kill        # ❌ 동작하지 않는다
lsof -ti :3000 | xargs kill  # ✅
```

파이프는 앞 명령의 출력을 뒤 명령의 **표준 입력(stdin)**으로 연결합니다. 그런데 `kill`은 stdin을 읽는 프로그램이 아니라 **인자(argument)**로 PID를 받는 프로그램입니다. 입력을 받는 통로가 다른 것입니다. `xargs`(**eXecute with ARGuments**)가 하는 일이 바로 이 변환입니다 — stdin으로 흘러온 텍스트를 다음 명령의 인자로 바꿔서 실행해 주는 **어댑터**입니다.

"이 명령은 stdin을 읽는가, 인자를 받는가"를 구분하기 시작하면 파이프라인 조립에서 헤매는 일이 급격히 줄어듭니다.

## 암호 같은 이름들의 어원

유닉스 명령어가 불친절해 보이는 이유는 1970년대의 산물이기 때문입니다. 전송 속도가 느린 물리 단말기에서 한 글자라도 아끼려던 문화가 이름에 남아 있습니다. 어원을 알면 무의미한 문자열이 기억 가능한 구조로 바뀝니다.

| 명령어      | 어원                                                                     |
| ----------- | ------------------------------------------------------------------------ |
| `lsof`      | **L**i**S**t **O**pen **F**iles                                          |
| `grep`      | 옛 편집기 ed의 명령 **g**/**r**egular **e**xpression/**p**rint           |
| `cd`, `pwd` | **c**hange **d**irectory, **p**rint **w**orking **d**irectory            |
| `pkill`     | **p**rocess kill — PID 대신 이름으로 시그널 전송 (`pkill -f "next dev"`) |
| `sudo`      | **s**uper**u**ser **do**                                                 |
| `ssh`       | **s**ecure **sh**ell                                                     |
| `xargs`     | e**X**ecute with **arg**ument**s**                                       |

## 왜 50년 된 시스템을 아직도 쓰는가

1969년에 태어난 유닉스의 후손 위에서 2020년대의 개발이 이루어지고 있다는 사실은, 생각해 보면 설명이 필요한 일입니다. 크게 네 가지 구조적 이유가 있습니다.

**1. 철학이 기술보다 오래 살아남았습니다.** "모든 것은 파일", "하나의 도구는 하나의 일만", "텍스트로 소통"이라는 추상화는 특정 하드웨어나 언어에 묶여 있지 않습니다. 구현은 몇 세대가 갈렸어도 좋은 추상화는 낡지 않았습니다.

**2. 이식성을 갖고 태어났습니다.** 유닉스는 C 언어와 함께 만들어진, 사실상 최초의 "다른 하드웨어로 옮길 수 있는 OS"였습니다. 어셈블리로 기계마다 다시 짜야 했던 시대에 이것은 결정적인 차이였습니다.

**3. 표준(POSIX)이 되었습니다.** 특정 회사의 제품이 아니라 규격이 되면서, 회사의 흥망과 무관하게 생태계가 존속할 수 있게 됐습니다.

**4. 지금 쓰는 것이 전부 유닉스 후손입니다.** macOS는 BSD 계열 유닉스이고, 서버·클라우드·컨테이너의 표준인 리눅스는 유닉스 호환 클론이며, iOS와 Android도 그 위에 서 있습니다. Docker 컨테이너 안도, CI 러너도, WSL도 전부 리눅스입니다. "백엔드/인프라를 배운다"는 말의 상당 부분은 "유닉스 환경을 다루는 법을 배운다"와 동의어입니다.

여기에 40년치 도구, 문서, 인력의 네트워크 효과가 얹혀 있어, 대체 시도들은 번번이 생태계의 관성을 이기지 못했습니다.

## 그래서, 어디까지 배워야 하나

이 지점에서 흔한 불안이 있습니다. "이런 명령어를 다 외우고 있어야 하나?" 결론부터 말하면 — **아무도 다 외우고 있지 않습니다.** 10년차 엔지니어도 `tar`와 `find`의 옵션은 매번 검색합니다. 연차가 만드는 차이는 암기량이 아니라, 에러를 봤을 때 원인 후보를 두세 개로 좁히는 **모델의 해상도**와 검색의 정확도입니다.

외울 것은 명령어가 아니라 개념입니다.

- 포트는 한 번에 한 프로세스만 점유한다 — 서버가 안 뜨면 점유자를 찾는다
- 프로세스는 PID로 식별되고, 시그널로 제어한다
- 도구들은 텍스트를 주고받으며, 파이프로 조립된다

셸 스크립트도 같은 기준으로 정리됩니다. **"읽고 고칠 수 있는 수준"은 필수, "능숙하게 짜는 수준"은 선택**입니다. 읽기가 필수인 이유는 분명합니다 — `package.json`의 scripts, Dockerfile의 RUN, CI 설정의 step, README의 설치 명령이 전부 셸 명령입니다. 읽지 못하면 이해 없는 복붙만 남습니다.

읽기 수준에 필요한 것은 사실상 네 가지 패턴이 전부입니다.

```bash
lsof -ti :3000 | xargs kill       # |     — 앞의 출력을 뒤의 입력/인자로
pnpm build && pnpm start          # &&    — 앞이 성공해야 뒤를 실행
pnpm dev > dev.log 2>&1           # >     — 출력을 파일로 (2>&1: 에러 출력까지)
DATABASE_URL="..." pnpm migrate   # VAR=  — 이 명령에만 적용되는 환경변수
```

반대로 본격적인 bash 프로그래밍(`if [[ ]]` 분기, 배열, 함수)은 깊이 파지 않아도 됩니다. 20줄을 넘는 자동화라면 익숙한 범용 언어로 짜는 편이 유지보수에 낫고, 이런 스크립트는 AI가 잘 생성해 주는 영역이기도 합니다 — 단, **생성된 것을 검증하려면 결국 읽을 줄 알아야 한다**는 점이 읽기 능력의 가치를 오히려 끌어올립니다.

<Callout type="tip" title="학습법 하나">
  강의로 명령어를 몰아서 배우는 것보다, 프로젝트를 하다 마주친 명령을 그
  자리에서 해부하는 편이 훨씬 오래 남습니다. 처음 보는 명령은{" "}
  <code>man 명령어</code>나 커뮤니티가 정리한 <code>tldr 명령어</code>로
  확인하는 습관이면 충분합니다.
</Callout>

터미널은 결국 50년간 다듬어진 하나의 사고방식 — 자원을 파일로 보고, 작은 도구를 텍스트로 잇는 — 위에 서 있습니다. 그 모델을 한 번 세워두면, 명령어는 암기의 대상이 아니라 모델에서 유도되는 결과가 됩니다.
