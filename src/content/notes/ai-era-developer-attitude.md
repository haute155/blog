> 이 글은 완성된 포스트가 아니라 **자라는 문서**입니다. 경험과 생각이 바뀔 때마다 업데이트되며, 변경 이력은 문서 하단 업데이트 로그에 남깁니다.

## 1. 코드 생성기가 아니라 작업자에게 절차를 맡겨라

### 원칙

AI에게 "코드 짜줘"가 아니라 **작업의 전체 절차**를 맡긴다. AI는 코드 생성기가 아니라 작업자·검증자·리뷰어다.

### 이유

- 브라우저 에러를 사람이 복붙해주던 일도, "네가 직접 브라우저를 열고 확인해"라고 시키면 AI가 스스로 한다. **문제는 AI의 능력이 아니라 사람의 상상력이었다.** "사용자님이 그렇게 시킨 적은 없었습니다."
- AI는 할 수 있어도 시키기 전까지는 먼저 하지 않는다. 물어보면 대부분 "됩니다"라고 답한다.
- 질문을 바꿔야 한다: "내가 이걸 할 수 있을까?" → **"어떻게 하면 얘한테 하게 만들 수 있을까?"**

### 실행 예시

```
나쁜 사용법:
  이 코드 고쳐줘.

좋은 사용법:
  1. 이슈를 먼저 정리해.
  2. 관련 파일을 찾아.
  3. 수정 계획을 세워. (내가 승인하기 전 구현 금지)
  4. 구현해.
  5. 타입 체크, 린트, 테스트를 돌려.
  6. 브라우저에서 직접 열어서 확인해. (Playwright 등)
  7. 변경 요약과 위험 지점을 남겨.
```

- 반복되는 절차는 커맨드/스킬로 고정한다: `/discuss`(의도 정렬) → `/plan` → 구현 → `/verify` → `/retrospect`
- 뭔가 귀찮은 일이 생길 때마다 한 번씩 물어본다: "이것도 네가 할 수 있어? 자동화할 수 있어? 매번 기억하게 할 수 있어?"

## 2. 암묵지를 명세화하라 — 나를 클로닝하기

### 원칙

내가 일할 때 무의식적으로 하는 판단과 절차를 밖으로 꺼내 AI가 따라 할 수 있는 문서·커맨드·스킬로 만든다.

### 이유

- 습관적으로 하던 일을 말로 푸는 것은 생각보다 어렵다. 그러나 이 과정 자체가 자기 실력의 재발견이다.
- 같은 AI라도 내가 절차를 정리하고 기준을 넣으면 결과가 달라진다. **"나로 인해 AI가 더 나아진다"는 감각을 한 번은 몸으로 겪어봐야 한다.**
- 커맨드를 만들기 어려우면 "커맨드를 만드는 커맨드"부터 만들면 된다 (메타 접근).

### 실행 예시

```
내 FE 작업 기준 (예시 — 자기 것으로 바꿀 것)

1. 요구사항이 모호하면 바로 구현하지 않는다.
2. 상태값의 출처(서버/클라이언트/임시)를 먼저 확인한다.
3. API 실패/로딩/빈 값/권한 없음 케이스를 확인한다.
4. 버튼은 disabled 조건과 클릭 가능 조건을 분리해서 본다.
5. 디자인과 다르게 구현한 부분은 이유를 남긴다.
6. 구현 후 모바일/반응형/접근성/키보드 포커스를 확인한다.
7. 작업 끝나면 변경 파일, 위험 지점, 추가 확인 필요 사항을 요약한다.
```

- 이 문서를 CLAUDE.md / 시스템 프롬프트 / 스킬에 넣는다.
- 실패한 작업은 버리지 말고 회고시킨다: 어떤 판단이 틀렸는지, 어떤 절차가 빠졌는지, 다음 기준에 무엇을 추가할지.

## 3. 주사위 게임 — 뭐가 6인지 아는 눈을 길러라

### 원칙

AI와의 작업은 주사위 게임과 같다. 잘 나온 것(6)은 고정하고, 빗나간 것만 골라 다시 던진다. **AI를 잘한다는 것은 좋은 프롬프트 한 방이 아니라, 결과물에서 무엇이 맞고 무엇이 틀렸는지 알아보는 능력이다.**

### 이유

- LLM은 스스로 "이건 6이고 이건 1입니다"라고 알려주지 않는다.
- 모르면 전부 다시 던지면 되지만 비용이 크다. 잘 뜬 6까지 흔들고, 다시 던져야 할 것을 고정해버린다.
- 프론트엔드는 특히 실패 신호가 흐리다. 빌드도 테스트도 통과하는데 제품은 이상한 상태가 계속 생긴다. "자연스럽게", "우리 제품답게" 같은 모호한 요구를 **실행 가능한 기준으로 고정**하지 않으면 결과물이 발산한다.

### 실행 예시

- 결과물을 받으면 축별로 평가한다: 디자인 / 상태 / 인터랙션 / 접근성 / 테스트 / 구조 — 각각 6인지 1인지.
- 다시 시킬 때는 범위를 좁혀 지시한다: "전부 다시" ❌ → "선택 상태 스타일만, 나머지는 유지" ✅
- 모호한 감각을 기준으로 번역하는 연습: "덜 튀게" → "배경색 채도를 낮추고 border 제거, 애니메이션 duration 유지"

## 4. 될 때까지 하는 집요함 — 수렴은 사람이 시키는 것

### 원칙

좋은 결과물은 한 번에 나오지 않는다. 무엇이 부족한지 보고, 원하는 것과 어떻게 다른지 말하고, 원하는 수준이 나올 때까지 계속 밀어붙인다.

### 이유

- "그냥 될 때까지 하니까 되던데요. 어떨 때는 기능 하나에 1주일도 걸렸어요." — 좋은 프로토타입을 만든 기획자의 답. 특별한 마법이 아니라 집요함이었다.
- AI가 모두에게 주어져도 결과물은 평준화되지 않는다. 어떤 사람은 평균적인 AI 화면에서 멈추고, 어떤 사람은 계속 더 밀고 간다.
- 좋은 환경(하네스)도 중요하지만, **결국 수렴은 사람이 시키는 것이다.**

### 실행 예시

```
작업 순서 (개발자의 코드 강박이 결과물을 막지 않게)

1단계: 일단 결과물을 만든다.
2단계: 마음에 들 때까지 감각을 밀어붙인다.
3단계: 그다음 코드 구조를 정리한다.
4단계: 제품으로 갈 수 있게 테스트/상태/접근성을 붙인다.
```

- "이 정도면 됐나?" 하고 넘기지 않는다. 마음에 안 들면 마음에 안 든다고 말하고 다시 시킨다.
- 한 번에 안 된다고 "역시 안 되네"로 결론 내리지 않는다. 무엇이 부족한지 → 방향 재지시 → 기준 추가 → 반복.

## 5. 맥락과 판단을 보존하라 — 걸어 다니는 히스토리의 종말

### 원칙

직접 코딩할 때 몸에 새겨지던 히스토리(왜 이 파일을 고쳤는지, 어떤 흐름이 위험한지)는 AI에게 맡기면 자동으로 쌓이지 않는다. **문서·테스트·이슈·보고서로 별도로 붙잡아야 한다.**

### 이유

- 코드는 빠르게 생기는데, 코딩하면서 같이 흡수하던 판단들은 흘러가버린다.
- 커밋이 쌓이는 것과 제품이 나아가는 것은 다르다. 하루 100커밋, 테스트 700개가 쌓여도 제품이 좋아진다는 보장은 없다.
- AI 시대의 개발에서는 **코딩 능력보다 맥락과 판단을 보존하는 기술이 더 중요해졌다.** 이것은 관리 업무가 아니라 개발을 잘하기 위한 핵심 작업이다.

### 실행 예시

- 작업 끝나면 md로 남기게 한다: 무엇을 바꿨는지, 왜 그렇게 했는지, 어떤 문제가 남았는지.
- 주요 흐름은 테스트로 고정한다 (구현 확인이 아니라 **사용자 흐름 보장** 기준으로).
- 의사결정 기록: 왜 이 구조를 선택했는지, 무엇을 시도했다 버렸는지.
- 문서가 감당 안 되게 쌓이면? → 문서를 보는 도구를 만들면 된다 (원칙 7).

## 6. 하네스 엔지니어링 — AI가 실수할 수 없는 환경을 만들어라

### 원칙

AI 자체를 똑똑하게 만드는 것(에이전트)과 별개로, **AI 외부의 작업 환경을 설계(하네스)**한다. 모든 하네스 구성 요소는 "모델이 단독으로 할 수 없는 것"에 대한 가정을 인코딩한다.

### 이유

- 프레임워크·라이브러리·린트 규칙·타입 시스템도 하네스다. AI 학습 데이터의 대부분이 이런 도구 위에 있으므로, 좋은 레이어를 만드는 것이 곧 AI 성능을 높이는 일이다.
- 예: 상태 관리 라이브러리 사용 시 eslint 규칙으로 AI의 흔한 실수를 차단하고, 나아가 **"실수할 수 없는 구조"로 라이브러리 자체를 재설계**할 수 있다.
- AI가 잘 달릴 수 있게 길을 닦는 일: 컨벤션, 구조 정리, 테스트, 나쁜 패턴 차단, 수정 가능한 단위로 코드 쪼개기.

### 실행 예시

- AI가 반복하는 실수를 발견하면 → 지적으로 끝내지 말고 **린트 규칙/타입/훅으로 구조화**한다.
- "끝나면 테스트는 무조건 돌려" 같은 필수 절차는 hooks로 강제한다 (기억에 의존하지 않는다).
- 회사 컨벤션을 시스템 프롬프트/CLAUDE.md로 인코딩한다.
- 검증 시스템을 만든다: AI의 산출물이 통과해야 하는 관문을 코드로 정의.

## 7. 큰 문제를 잡아라, 막히면 도구를 만들어라

### 원칙

작은 문제에서는 아무것도 드러나지 않는다. 엄두가 안 나던 크기의 문제를 AI와 함께 굴려보고, 막히면 그 막힘을 푸는 도구를 만든다.

### 이유

- 문제가 커져야 코딩 바깥의 개발(요구사항, 문서, 테스트, 검수 기준)이 드러난다.
- 시작의 허들이 낮아졌다. 예전에는 비용 계산만 하다 접었던 아이디어를 이제 "일단 첫 형태"까지 만들어볼 수 있다. 질문이 바뀐다: "이걸 내가 만들 수 있을까?" → **"일단 어디까지 만들어볼 수 있을까?"**
- 완제품이 필요한 게 아니다. Obsidian을 다시 만드는 게 아니라 **지금 내 문제를 풀 만큼의 md 뷰어**를 만들면 된다. 예전엔 배보다 배꼽이 컸던 우회로가 실제 선택지가 되었다.

### 실행 예시

- 코드가 안 보이면 코드 분석 도구를, 흐름이 안 보이면 캔버스 뷰어를, 디버깅이 답답하면 devtool을 만든다.
- 채팅과 IDE 밖으로 시야를 넓힌다: GitHub(이슈/PR/Wiki로 일을 나누고 검수하는 구조) → Notion, Calendar, Slack, Sheets까지. AI가 만질 수 있는 범위는 코드 파일보다 훨씬 크다.

## 8. 실력이 녹슬지 않게 — 판단의 근육을 유지하라

### 원칙

AI가 struggle(고생)을 없애면 학습 루프도 사라진다. **의도적으로** 진짜 실력이 쌓이는 습관을 유지한다.

### 이유

- 전통적 성장 루프는 "만들고-깨지고-고치는" 과정에서 직관과 취향(taste)이 쌓이는 구조였다. AI가 즉시 동작하는 코드를 주면 이 루프가 끊긴다.
- 이해 없이 출고하는 코드가 쌓이면, AI 산출물을 평가할 기준 자체가 자라지 않는다.
- 판단력(원칙 3)의 원천은 결국 직접 겪은 문제들이다. "직접 풀어본 모든 문제가 나중에 직관이 끌어다 쓸 재료가 된다."

### 실행 예시

- **AI와 진짜 대화하기**: 생성된 코드에 대해 트레이드오프, 대안, 이유를 되묻는다. 트랜잭션이 아니라 멘토처럼 쓴다.
- **의도적으로 느려지기**: 실행 전 몇 분이라도 프롬프트와 엣지케이스를 먼저 생각한다.
- **정기적으로 손으로 짜기**: AI 없이 문제를 풀어 평가 감각을 유지한다.
- **남의 코드 많이 읽기**: 시니어의 PR, 좋은 오픈소스. 단, 그럴싸해 보이는 AI 생성 코드를 무비판적으로 흡수하지 않도록 주의.
- 자신에게 정직하기: **내가 이해한 것과 그냥 복붙한 것을 구분한다.**

## 9. 자기 포지션을 정하라 — 개발자는 하나로 수렴하지 않는다

### 원칙

AI 시대의 개발자는 하나의 모습으로 수렴하지 않고 더 여러 갈래로 갈라진다. 자기가 잘 보는 것이 무엇인지 파악하고 의식적으로 포지션을 정한다.

### 이유

- 유튜브가 영상 제작의 허들을 낮추자 역할이 다양해졌듯(출연/기획/편집/채널운영/플랫폼), 개발도 같은 방향으로 간다.
- 기획자·디자이너·PM이 직접 프로토타입을 만들어오는 시대. 모든 결과물이 개발자에게서 시작하지 않는다.
- 코드가 눈에 밟히는 감각은 약점(속도를 늦춤)이자 강점(결과물을 더 멀리 보냄)이다. **결과물을 밀어붙이던 사람도 어느 순간 구조의 벽에서 막히고, 그때 길을 닦아주면 다시 움직인다.**

### 갈래들

| 포지션                        | 하는 일                                                                          |
| ----------------------------- | -------------------------------------------------------------------------------- |
| **밀어붙이는 사람**           | 문제를 가장 잘 아는 사람으로서 될 때까지 수렴시켜 결과물을 뽑는다                |
| **닦는 사람**                 | 가능성 있는 결과물이 버려지지 않고 제품으로 이어지게 구조·맥락·테스트를 정리한다 |
| **레이어 만드는 사람**        | AI와 사람이 더 잘 만들 수 있는 도구·컴포넌트·플랫폼·하네스를 만든다              |
| **깊은 시스템을 만드는 사람** | 여전히 필요한 복잡한 제품의 설계·운영·성능·보안                                  |

### 실행 예시 — "닦는 사람"이 던지는 질문들

```
이 버튼은 언제 비활성화돼?
API 실패하면 사용자는 뭘 봐?
권한이 없으면 페이지 접근은 막아?
모바일에서는 이 인터랙션이 가능한가?
이 상태는 서버 상태야, 클라이언트 임시 상태야?
이 프로토타입을 제품 코드로 바꾸려면 어디를 정리해야 해?
```

"대충 돌아가는 화면"은 앞으로 넘쳐난다. 그것을 진짜 서비스로 만들려면 누군가는 이 질문을 해야 한다.

## 10. 증명 방식을 바꿔라

### 원칙

채용 기준이 올라간 시대의 증명은 코딩 테스트 점수가 아니라 **"AI와 함께 무엇을 끝까지 만들어냈는가"**다.

### 이유

- 같은 모델, 같은 도구, 같은 기능을 써도 사람마다 결과가 너무 다르다. 차이는 사고의 범위와 경험이다: 어디까지 시켜볼 수 있다고 상상하는지, 무엇을 맡기고 무엇을 직접 봐야 하는지, 막혔을 때 어디까지 우회하는지, 마음에 안 들 때 무엇을 다시 요구하는지.
- 데모와 제품 사이의 간격을 실제로 건너본 기록이 가장 강한 포트폴리오다.

### 실행 예시

- 사이드 프로젝트 하나를 데모가 아니라 **제품 수준까지** 밀어본다 — 문서, 테스트, 엣지케이스, 접근성까지.
- 그 과정에서 만든 스킬/하네스/도구와 시행착오를 글로 남긴다 (방법론은 금방 낡지만, **경험의 기록**은 남는다).

- ❌ AI의 "다 했습니다"를 그대로 믿기 — 검수 없는 신뢰는 신뢰가 아니다 (→ 원칙 3)
